## Проблема

База данных уходила в глубокие блокировки при обновлении остатков. Мобильное приложение CRM показывало пустые экраны или ошибки доступа — из-за бага в маршрутизации многосайтовости ядро путало публичный домен магазина с доменом внутреннего портала. Вдобавок AI-боты нейросетей агрессивно сканировали умный фильтр и исчерпывали все воркеры веб-сервера.

## Решение

Полный аудит и миграция на высокопроизводительное хранилище. Хранение сессий перенёс в оперативную память, чтобы убрать блокировки при параллельных AJAX-запросах:

```ini
session.save_handler = redis
session.save_path = "tcp://127.0.0.1:6379"
```

Жёсткое разделение многосайтовости прямым SQL:

```sql
UPDATE b_lang SET SERVER_NAME='crm.example.com' WHERE LID='s1';
UPDATE b_lang SET SERVER_NAME='example.com' WHERE LID='s2';
```

Строгая фильтрация агрессивных AI-парсеров на уровне Nginx:

```nginx
if ($http_user_agent ~* "meta-externalagent|FacebookBot|Bytespider") { return 403; }
```

Единый кеш для витрины и портала за счёт статического идентификатора сессии.

- Мобильная CRM полностью восстановлена

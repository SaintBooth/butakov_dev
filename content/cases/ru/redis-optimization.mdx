## Проблема

Проект на Битриксе упёрся в лимиты железа хостинга. Redis занимал 977MB из доступного 1GB. `INFO memory` показывал критическое заполнение. Политика вытеснения LRU не справлялась с интенсивностью записи тегов. Стандартный ответ админов: `FLUSHDB` — но он уничтожает все ключи пользовательских сессий, мгновенно разлогинивает пользователей и очищает корзины. Первый утренний запрос к холодному кешу приводил к 10-секундной генерации страницы.

## Решение

Аудит Redis через `--bigkeys` выявил причину: 84% ключей — протухшие наборы тегов. Вместо полного сброса написал bash-скрипт поиска ключей по маске и асинхронного удаления через `UNLINK` (освобождает память без блокировки основного потока):

```bash
REDIS_CLI="/usr/bin/redis-cli"
$REDIS_CLI --scan --pattern '*IL:*' | xargs -L 100 $REDIS_CLI unlink
$REDIS_CLI --scan --pattern '*domain.com/public_html*' | xargs -L 100 $REDIS_CLI unlink
```

Поставил в cron на часы минимальной нагрузки. Добавил модуль прогрева кеша — сервер читает sitemap и запрашивает целевые URL через curl до прихода реальных посетителей:

```bash
urls=$(curl -s https://domain.com/sitemap.xml | grep -oP '(?<=<loc>).*?(?=</loc>)' | head -n 10)
for url in $urls; do
  curl -s -L -o /dev/null "$url"
done
```

## Результат

- **Освобождено 97% RAM** — потребление упало с 688MB до 19MB
- Пользовательские сессии полностью сохранены при очистке
- Ответ сервера для первых посетителей сокращён до миллисекунд
- Остались на текущем тарифе хостинга — апгрейд железа не понадобился

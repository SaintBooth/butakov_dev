Вводные проекта такие. Заказчик - бренд женской одежды из Екатеринбурга. Товарный учёт ведётся в 1С:Розница. Дизайн уже готов - React-прототип в эстетике quiet luxury, его нужно воспроизвести один в один. Бюджет небольшой: без лицензий за сто тысяч в год и без команды из шести человек. Один разработчик, open-source стек, ограниченные сроки.

Отсюда стек: WordPress + WooCommerce, блочная тема с нуля, обмен с 1С по протоколу CommerceML. Вторая особенность проекта - сам процесс: практически весь код написан в паре с Claude Code. Вайб-кодинг - это когда разработчик ставит задачу и проверяет результат, а рутинный код пишет AI-агент. В демках это выглядит как магия. На реальном проекте с 1С, вариативными товарами и 152-ФЗ это обычная инженерная работа, просто заметно быстрее.

## Проблема: бутик-магазин без бутик-бюджета

В России разработка магазина с учётом в 1С почти всегда сводится к совету «бери Битрикс». Поисковая выдача это подтверждает: по запросам про интеграцию 1С с сайтом почти вся семантика битриксовая. Логика понятна - штатный модуль обмена, коробочные тарифы, много интеграторов.

Для небольшого бренда одежды эта логика ломается о деньги и о дизайн.

Деньги. Лицензия «1С-Битрикс: Малый бизнес», ежегодные продления, работа интегратора - в сумме это бюджет, сопоставимый со всей разработкой на open source. Для бутика с одной розничной точкой такие расходы не оправданы.

Дизайн. Заказчику нужен не «шаблон с каталогом», а точное воспроизведение прототипа: четыре цвета в палитре, типографика Lora/Raleway, нулевые радиусы, воздушная сетка. Натягивать такое на готовую тему дольше, чем написать свою.

При этом требования никуда не деваются: реал-тайм остатки из 1С:Розница (одежда - это вариативные товары, у каждого размера и цвета свой сток), ЮKassa со СБП, доставка СДЭК, cookie-баннер и согласия на обработку ПД по 152-ФЗ, ecommerce-разметка Яндекс.Метрики.

Итого: функциональность взрослого магазина, минимальный бюджет, кастомный дизайн - и всё это одновременно. Ставка была на две вещи: бесплатный стек WordPress + WooCommerce и AI-агент вместо команды.

## Решение

Стек проекта целиком, дальше по каждому пункту подробно:

- CMS: WordPress, полностью блочная тема (FSE) с нуля, без page-builder-плагинов
- E-commerce: WooCommerce + WooCommerce Blocks, мини-корзина через Store API
- Бэкенд: PHP 8, модульная структура `inc/`, ACF (поля регистрируются кодом, не через UI)
- Фронтенд: ES-модули без jQuery, `IntersectionObserver` для анимаций, `history.pushState` для состояния фильтров, иконки Lucide инлайн SVG-спрайтом
- Дизайн-токены: `theme.json` как единый источник, CSS-переменные генерируются из него
- Обмен с 1С: 1С:Розница, протокол CommerceML 3.1, реал-тайм
- Платежи и сервисы: ЮKassa (карты, СБП), СДЭК, Unisender, Яндекс.Метрика с ecommerce-событиями
- Хостинг и деплой: NetAngels, деплой по SSH
- AI-инструменты: Claude Code + MCP-серверы (Playwright для проверки в браузере, SSH для деплоя)

### Стек и архитектура: блочная тема как дизайн-система

Основа - WordPress с полностью блочной темой (FSE, Full Site Editing), без page-builder-плагинов. Для бюджетного проекта это принципиально: каждый конструктор типа Elementor - это потеря производительности, зависимость от чужих обновлений и лишний слой между дизайном и кодом.

Вместо этого дизайн-система из React-прототипа переехала в `theme.json` - единый источник правды для токенов. Палитра из четырёх цветов, шкала отступов на базе 8px, два шрифта. WordPress сам генерирует из этого CSS-переменные, и весь дальнейший CSS ссылается только на токены:

```css
:root {
  --color-bone: /* фон, ~70% площади */;
  --color-ink: /* текст и UI */;
  --color-bordeaux: /* sale и акценты */;
  --color-pine: /* тёмная навигация */;
}
```

Иерархия текста строится через прозрачные варианты основного цвета, а не через набор серых. Радиусы нулевые везде, кроме капсульных тегов. Этот жёсткий набор правил потом сыграл важную роль в AI-разработке, об этом ниже.

PHP-слой организован просто: `functions.php` ничего не делает сам, только подключает модули из `inc/`, у каждого файла одна ответственность:

```php
// functions.php: только подключение, никакой логики
require_once get_theme_file_path( 'inc/setup.php' );
require_once get_theme_file_path( 'inc/assets.php' );
require_once get_theme_file_path( 'inc/woocommerce.php' );
require_once get_theme_file_path( 'inc/ajax-handlers.php' );
```

Повторяющиеся секции вынесены в block patterns, обвязка типа шапки и футера - в template parts. WooCommerce-шаблоны переопределены точечно, только там, где стандартная разметка не совпадала с дизайном.

### AI-workflow: как выглядит вайб-кодинг на реальном проекте

Агент не написал магазин сам. Он написал большую часть кода, а я проектировал, ставил задачи и проверял результат. Работы не меньше, чем при ручной разработке, но пропускная способность другая.

Workflow держится на трёх вещах.

Контракт в CLAUDE.md. У Claude Code есть файл инструкций проекта, агент читает его каждую сессию. Туда я вынес всю дизайн-систему в виде запретов: палитра ровно из четырёх цветов, сырые hex-коды запрещены, радиус ноль, заголовки только uppercase с трекингом, отступы только из 8px-шкалы. LLM охотно импровизирует, если дать волю. Жёсткие правила эту импровизацию убирают: когда агенту нужен новый оттенок, он не вставляет hex, а выводит его через `oklch()` от существующего токена, потому что так написано в правилах.

Файловая память между сессиями. Контекст агента конечен, а проект жил месяцы. Каждый содержательный урок («в WooCommerce Blocks не грузится jQuery», «дельта-выгрузки 1С стирают вариации») записывался в markdown-файлы памяти. Новая сессия начинается не с нуля, а с конспекта всех предыдущих граблей. По сути это внешняя долговременная память инженера, только читает её не человек, а агент.

Проверка браузером, а не «на глаз». У агента есть доступ к Playwright через MCP: он сам открывает страницу, кликает, снимает скриншот и сравнивает с прототипом. Правило простое: вывод «баг починен» не делается без живой проверки. Оно окупилось на истории с карточками товаров. Клики по ссылкам не работали только на десктопе и только на проде. Агент воспроизвёл баг реальным Playwright-кликом и нашёл причину: `setPointerCapture` в карусели перехватывал pointer-события и ретаргетил клик со ссылки на трек карусели. На глаз этот диагноз поставить было нельзя, код карусели выглядел корректным.

Что агент делает плохо. Уверенно «чинит» не ту причину, если не заставить его сначала воспроизвести баг. Тащит в вёрстку решения мимо дизайн-системы, если правила размыты. Теряет контекст на длинных задачах без внешней памяти. Все три проблемы решаются процессом, поэтому в статье столько внимания процессу, а не только коду.

### Интеграция с 1С: CommerceML и грабли, о которых молчат мануалы

Самая дорогая по времени часть проекта. И самый полезный раздел для тех, кто ищет интеграцию 1С с сайтом не на Битриксе.

Обмен построен на протоколе CommerceML: 1С:Розница по расписанию и по событию (включён реал-тайм обмен) выгружает XML-пакеты, плагин на стороне WordPress их разбирает и обновляет каталог. Схема простая, грабли начинаются в деталях.

Грабля 1: остатки, которых не было. Симптом: на витрине висели неверные остатки, хотя обмен отрабатывал без ошибок. Причина нашлась только диффом сырого XML с тем, что доехало до базы. В CommerceML 3.1 остатки приходят отдельными файлами `rests_*.xml` со вложенной структурой, остаток разбит по складам внутри `<Остатки><Остаток><Склад>`:

```xml
<Предложение>
  <Ид>guid-товара#guid-характеристики</Ид>
  <Остатки>
    <Остаток>
      <Склад>
        <Ид>guid-склада</Ид>
        <Количество>3</Количество>
      </Склад>
    </Остаток>
  </Остатки>
</Предложение>
```

Парсер плагина ждал плоский `<Количество>` уровнем выше и молча пропускал вложенный вариант. Ошибок нет, предупреждений нет, обмен выглядит рабочим, а данные при этом неверные. Лечилось патчем парсера. С тех пор перед обновлением 1С-плагина я диффаю его код: следующий апдейт может молча вернуть старое поведение.

Грабля 2: два независимых пайплайна атрибутов. В CommerceML свойства номенклатуры («ЗначенияСвойств») и характеристики вариаций («ХарактеристикиТовара») - разные механизмы, плагин обрабатывает их разными путями. Цвет, заведённый как свойство, и размер, заведённый как характеристика, приезжают в WooCommerce по-разному. Пока не разложишь оба пайплайна по шагам, результат непредсказуем.

Грабля 3: дельта-выгрузки против вариаций. Быстрый обмен шлёт не полный каталог, а дельты-чанки. Плагин на каждый чанк пересоздавал набор вариаций товара, и вариации, не попавшие в текущий чанк, просто исчезали. Размерная сетка постепенно пустела. Решение - режим сохранения вариаций при частичных выгрузках плюс чистка базы: за время отладки скопилось около тысячи вариаций-сирот.

Грабля 4: GUID вместо слагов. Термины атрибутов создавались со слагами-GUID, это ломало URL фильтров и читаемость админки. Решилось константами соответствия и repair-скриптом, который прогоняется после обмена.

Вывод по разделу. Интеграция 1С с WordPress работает стабильно, у нас в проде реал-тайм обмен. Но это не «поставил плагин и забыл». Придётся читать код плагина, править парсер и писать проверочные скрипты. Закладывайте это в оценку.

### Платежи, доставка и 152-ФЗ

Российский контур собирался из стандартных блоков: ЮKassa (карты и СБП) на приёме платежей, СДЭК и Почта России на доставке, Unisender на рассылках, ecommerce-события Яндекс.Метрики на аналитике. Отдельная статья расходов бюджетного проекта, про которую забывают в оценках, - 152-ФЗ: cookie-баннер, чекбоксы согласия на обработку персональных данных в формах, оферта в чекауте. Технически несложно, но такие задачи всплывают в последнюю неделю и двигают срок, если не заложены с самого начала.

### Фронтенд без jQuery: сюрприз от WooCommerce Blocks

Современный WooCommerce с блочными шаблонами не грузит jQuery на витрину. Вообще. Классические сниппеты из туториалов вида `$('.variations_form').on('found_variation', …)` просто не с чем запускать. Для блочной темы пришлось написать нативный резолвер вариаций: собрать выбранные атрибуты, найти совпадение в данных вариаций, обновить цену, фото и доступность:

```js
// jQuery отсутствует: WooCommerce Blocks его не подключает.
const match = variations.find((v) =>
  Object.entries(selected).every(
    ([attr, value]) => !v.attributes[attr] || v.attributes[attr] === value
  )
);
```

Из той же серии: AJAX-фильтры каталога рендерятся внутрь блочного шаблона товаров, состояние фильтров живёт в URL через `history.pushState` (ссылку с выбранными фильтрами можно переслать), мини-корзина работает через Store API. Скролл-анимации на `IntersectionObserver` с учётом `prefers-reduced-motion`. Ничего необычного, просто современный фронтенд, который в экосистеме WordPress до сих пор приходится собирать руками.

## Результат

Что в проде на момент публикации.

Работающий магазин на кастомной блочной теме, которая воспроизводит дизайн-прототип. Каталог с AJAX-фильтрами, вариативные карточки с размерами и цветами, избранное, страница товара со свайп-галереей на мобильных.

Реал-тайм обмен с 1С:Розница: цены, остатки и вариации приезжают из товароучётки без ручного дублирования. После фикса парсера остатков витрина совпадает со складом.

Российский контур: приём платежей через ЮKassa/СБП, доставка СДЭК и Почтой России, требования 152-ФЗ закрыты.

Скорость итераций: два полных пакета правок заказчика, тридцать пунктов от типографики до перестройки блоков, разобраны за несколько дней каждый. Время разработки считалось неделями, не месяцами.

Бюджет: нулевая стоимость лицензий. WordPress, WooCommerce и весь используемый стек - open source. Деньги заказчика ушли на работу и хостинг, а не на ежегодные продления коробки.

### Когда WordPress + AI подходит, а когда нет

Подходит, если у вас малый или средний магазин с кастомным дизайном, учёт в 1С без глубокого ERP-контура, бюджет критичен и есть разработчик, готовый разбираться в XML обмена, а не только ставить плагины.

Не подходит, если у вас высоконагруженный маркетплейс с тысячами заказов в день, сложная логистика с резервированием по складам в реальном времени или корпоративные требования к вендорской поддержке. В таких случаях коробка или кастом на другом стеке оправданы.

Про AI-часть вывод такой. Claude Code не заменяет разработчика, он заменяет часть команды. Проект такого объёма в одиночку и в такие сроки без агента я бы не сделал, либо сделал бы с другим бюджетом. Но без выстроенного процесса (правила в CLAUDE.md, внешняя память, обязательная проверка браузером) агент выдаёт уверенный, но неверный код. Настроить процесс важнее, чем научиться писать удачные промпты.

## FAQ

**Сколько стоит разработка интернет-магазина на WordPress?**
Порядок цифр в разы ниже разработки на коробочных платформах: нет стоимости лицензий, меньше ручной работы за счёт AI-разработки. Точная цена зависит от дизайна, интеграций и каталога, считается по ТЗ.

**Можно ли интегрировать 1С с WooCommerce?**
Да, через протокол CommerceML, включая реал-тайм обмен остатками и вариативные товары. Нюанс: штатные плагины требуют доработки напильником, подробности в разделе про 1С выше.

**Что такое вайб-кодинг простыми словами?**
Это разработка, в которой рутинный код пишет AI-агент, а человек ставит задачи, проектирует архитектуру и проверяет результат. Скорость растёт кратно, качество держится на дисциплине проверок.

**Заменяет ли Claude Code разработчика?**
Нет. Он заменяет часть команды: джуна-верстальщика, тестировщика на рутинных проверках, техписателя. Архитектура, декомпозиция задач и приёмка остаются на человеке.

**Работает ли WooCommerce с ЮKassa и СБП?**
Да, у ЮKassa есть официальный модуль для WooCommerce с поддержкой оплаты картами и через СБП.

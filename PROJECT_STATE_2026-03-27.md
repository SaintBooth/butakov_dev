# Butakov Site - текущее состояние

Дата фиксации: 2026-03-27

## 1) Что это за проект

Одностраничный сайт-портфолио на `React + Vite` с Tailwind CSS v4 (через `@tailwindcss/vite`), без роутинга и без backend-части в этом репозитории.

Ключевая особенность текущей реализации: вся логика и контент страницы сосредоточены в одном компоненте `src/App.jsx`.

## 2) Технологический стек

- `react` `19.2.4`
- `react-dom` `19.2.4`
- `vite` `8.0.0`
- `@vitejs/plugin-react`
- `tailwindcss` `4.2.1`
- `@tailwindcss/vite`
- `lucide-react` (иконки)

Скрипты из `package.json`:

- `npm run dev` - локальная разработка
- `npm run build` - production-сборка в `dist`
- `npm run preview` - локальный просмотр production-сборки
- `npm run lint` - ESLint

## 3) Структура проекта (фактически используемая)

- `index.html` - точка входа
- `src/main.jsx` - монтирование `App`
- `src/App.jsx` - основная логика и весь UI сайта
- `src/index.css` - подключение Tailwind (`@import "tailwindcss";`)
- `public/` - статические файлы (`butakov-01.png`, `favicon.svg`, `icons.svg`)
- `dist/` - итог сборки для деплоя

Примечание: `src/App.css` сейчас выглядит как шаблонный/наследованный файл из стартового Vite-шаблона и, по текущей конфигурации, не используется в рантайме.

## 4) Как сайт работает сейчас

### 4.1 Рендер и навигация

- SPA без клиентского роутера: секции внутри одной страницы (`#services`, `#b2b`, `#experience`, `#promptspace`, `#cases`, `#blog`, `#process`, `#contact`).
- Навигация якорями и программный скролл до формы через `scrollToContact`.
- Есть мобильная нижняя навигация.

### 4.2 Данные и контент

Контент хранится прямо в `App.jsx` в массивах:

- `experienceData`
- `hardSkills`
- `softSkills`
- `portfolioCases`
- `blogPosts`
- `services`

Это ускоряет изменения на маленьком проекте, но увеличивает размер и сложность одного файла.

### 4.3 Форма заявки

- Отправка через внешний сервис `Web3Forms` (`https://api.web3forms.com/submit`).
- Ключ `access_key` зашит в клиентский код.
- После успешной отправки показывается блок подтверждения.

### 4.4 SEO и аналитика

- В `useEffect` программно выставляются `title` и часть meta-тегов.
- Подключена Яндекс.Метрика (`id=107722106`) с инициализацией в браузере.
- Добавлен JSON-LD (`ProfessionalService`) в runtime.

### 4.5 Модальные окна

- Модалка кейса
- Модалка статьи
- Модалка политики конфиденциальности

При открытии модалок блокируется скролл `body`.

## 5) Статическая сборка и деплой

Текущий рабочий pipeline (ручной):

1. Локально выполнить:
   - `npm install` (если нужно)
   - `npm run build`
2. Проверить, что `dist/` содержит:
   - `dist/index.html`
   - `dist/assets/index-*.css`
   - `dist/assets/index-*.js`
   - статические файлы из `public/`
3. Залить содержимое `dist/` на хостинг вручную.

Проверка на дату фиксации:

- Сборка успешна (`vite build`, ~3.28s).
- Артефакты присутствуют в `dist/`.

## 6) Операционная модель проекта

- Репозиторий локальный (без git в текущей рабочей папке).
- Деплой выполняется вручную, без CI/CD.
- Версия в проде фактически определяется последним загруженным набором файлов из `dist`.

## 7) Текущие риски и техдолг (без критики архитектуры)

1. Секреты в клиенте:
   - `Web3Forms access_key` находится в фронтенд-коде.
   - Для статического сайта это частая практика, но ключ публичный по сути.
2. Монолитный `App.jsx`:
   - Файл большой, сложнее поддерживать и тестировать.
3. Юридический текст:
   - Политика помечена как базовый шаблон (есть прямой комментарий в тексте).
4. Ручной деплой:
   - Нет автоматической валидации перед публикацией (lint/build/smoke по чек-листу зависят от дисциплины).

## 8) Минимальный регламент ручного релиза (рекомендованный)

Перед выкладкой в прод:

1. `npm run lint`
2. `npm run build`
3. `npm run preview` и ручная проверка:
   - открываются все секции;
   - работают модалки;
   - форма отправляется;
   - нет критичных ошибок в консоли;
   - ссылки и якоря корректны.
4. Загрузка `dist/` на сервер.
5. Быстрая smoke-проверка на прод-домене.
